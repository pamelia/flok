# 1 to 10 Implementation Spec

## Purpose

This document is the execution spec for finishing the ten major product/workstream upgrades that would move flok from "strong early coding agent" to "serious top-tier coding agent".

It is not a greenfield spec. It is a continuation spec built against the repo as it exists now.

It captures:

- the original intent behind each of the ten workstreams
- what is already implemented in this repo
- what is still missing
- the exact architecture and code paths that should be extended
- test and acceptance requirements
- sequencing, dependencies, and risks

This file should be treated as the operational source of truth for the remaining work on items `#1` through `#10`.

## Global Rules

These rules apply to every workstream below:

- Do not try to land all ten workstreams in one patch set.
- Keep the repo buildable after each feature slice.
- After every code-bearing slice, run the full gate:
  - `cargo fmt --all --check`
  - `cargo clippy --workspace --all-targets -- -D warnings`
  - `cargo test --workspace`
- Do not revert unrelated user changes in a dirty worktree.
- Prefer in-place incremental evolution over immediate crate extraction.
- Extract to new crates only after the behavior exists and is tested.

## Definition Of Done For The Overall 1 To 10 Program

The full program is done only when all of the following are true:

- Every workstream below meets its own acceptance criteria.
- Routing, planning, verification, teams, security, and LSP behavior operate together end to end.
- New behavior is visible to the TUI through explicit bus events.
- Partial state is durable across restart where the spec requires it.
- Unsafe or ambiguous agent behavior is rejected by runtime primitives, not left to prompting alone.
- All new code paths have unit and integration coverage.
- The only remaining failing tests are either fixed or explicitly quarantined with a justified issue and replacement coverage.

## 1. Plan/Execute Runtime

### Goal

Move from "stored structured plans plus a minimal sequential executor" to a real execution runtime with approval, resumability, rollback, checkpoints, and dependency-aware execution.

### Already Implemented

- Typed persisted plans in `crates/flok-core/src/plan.rs`
- JSON-backed plans under `.flok/plans`
- `plan_create` and `plan_update` tools in `crates/flok-core/src/tool/plan_tools.rs`
- basic plan listing/show/approve/execute wiring in `crates/flok-core/src/session/engine.rs`
- TUI commands for plan actions in `crates/flok-tui/src/app.rs`
- minimal sequential plan execution
- durable `PlanRun`, `PlanRunStep`, `PlanResumePoint`, and failure metadata persisted in plan JSON
- plan-created file fingerprints and execution-time stale-file pause before affected steps run
- run metadata updates for executing/completed/failed/cancelled plan execution paths
- `/rollback-plan` restores the selected or latest step checkpoint, marks the plan/run rolled back,
  and resets dependent steps so execution can resume safely

### Missing

- fuller durable execution state machine and remaining recovery commands
- full DAG scheduler
- resumable step execution after restart
- richer checkpoint rollback semantics and user-facing target selection
- stale-plan remediation flow after detection
- step-level approval mode
- user-visible plan diffing
- explicit plan failure recovery flows

### Required Architecture

Add explicit runtime entities:

- `PlanExecutionState`
- `PlanCheckpoint`
- `PlanRun`
- `PlanRunStep`
- `PlanResumePoint`

Recommended model:

```rust
pub enum PlanRunStatus {
    Draft,
    Approved,
    Executing,
    Paused,
    Failed,
    Completed,
    Cancelled,
    RolledBack,
}
```

Each `PlanStep` must have:

- stable `step_id`
- dependency list
- planned file scope
- execution prompt
- status
- checkpoint pointer
- started/completed timestamps
- retry count
- failure metadata

### File Touch Points

- `crates/flok-core/src/plan.rs`
- `crates/flok-core/src/session/engine.rs`
- `crates/flok-core/src/bus.rs`
- `crates/flok-core/src/snapshot.rs`
- `crates/flok-core/src/worktree.rs`
- `crates/flok-core/src/tool/plan_tools.rs`
- `crates/flok-tui/src/app.rs`

### Implementation Plan

1. Add durable run metadata to the plan JSON model first.
2. Introduce explicit executor state transitions instead of mutating only plan step status.
3. Add checkpoint creation before every mutating step.
4. Add rollback target selection:
   - rollback to step start
   - rollback full plan
5. Replace the current strictly sequential executor with a DAG-ready scheduler.
6. Gate parallel step execution behind a conservative config flag.
7. Add stale-plan detection by comparing planned files against current hashes.
8. Add `/resume-plan` and `/rollback` runtime commands.

### Acceptance Criteria

- approved plans can be resumed after process restart
- failed plans can stop at a step and leave a resumable state
- rollback restores files to the checkpoint before the failed or requested step
- stale plans are detected and surfaced before execution continues
- step dependency violations are impossible at runtime

### Test Requirements

- cycle rejection
- JSON persistence round-trip
- sequential execution
- dependency blocking
- parallel-ready scheduling behavior
- failure pause and resume
- rollback restores content correctly
- stale plan detection
- TUI command coverage for approve/show/execute/resume/rollback

## 2. Verification Loop

### Goal

Finish the verify/fix loop so the agent behaves like a disciplined engineer after every meaningful code change, not just a generator with a single retry.

### Already Implemented

- automatic post-edit verification
- ecosystem-aware command selection
- changed-file scoping for Rust, Python, Go, and runner-aware Node
- explicit verification events in the bus/TUI
- one bounded auto-repair retry
- verification scope reuse across retries
- failure-family classification
- retry churn detection
- retry scope relevance handling
- continuity handling across skipped non-verifying rounds
- `VerificationPolicy`, `VerificationRecord`, `VerificationLevel`, and `VerificationStopReason`
- durable plan-step verification records with command, scope, outcome, duration, failure kind,
  failure impact, and summary
- plan execution gates step completion on required verification passing or an explicit
  no-command skip reason
- red plan-step verification marks the step/plan failed and uses the existing rollback path
- `/show-plan` surfaces latest verification status for active run steps

- `crates/flok-core/src/bus.rs`

- verification presets by repo type
- smarter test targeting beyond current heuristics
- "fix until clean" bounded mini-loop for some categories
- broader session-level verification history beyond plan run metadata

### Required Architecture

Add:

- `VerificationPolicy`
- `VerificationRecord`
- `VerificationLevel`
- `VerificationStopReason`

Suggested levels:

- `Sanity`
- `Targeted`
- `RepoDefault`

4. Add stronger targeting:

5. Add bounded multi-step self-fix for failures classified as repairable.
